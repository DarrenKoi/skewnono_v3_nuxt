# 09. UI 용어 및 문구 가이드

이 문서는 `front-dev-home` UI를 한국 회사 내부 엔지니어가 사용하는 것을 전제로, 화면 용어와 문구 톤을 일관되게 맞추기 위한 기준입니다.

목표는 다음과 같습니다.

- 영어 직역보다 현업에서 바로 이해되는 표현을 사용합니다.
- 같은 개념은 화면마다 같은 한국어로 유지합니다.
- 라벨만 번역하지 않고, 클릭성과 가독성까지 함께 고려합니다.

## 1. 기본 원칙

### 1.1 사용자 기준

- 이 웹 앱의 기본 사용자는 한국 엔지니어입니다.
- 가능하면 UI는 한국어를 우선 사용합니다.
- 다만 장비명, 공정명, 제품명, 시스템 약어처럼 현업에서 영어/약어가 더 익숙한 항목은 그대로 유지합니다.

예시:

- `CD-SEM` -> `CD-SEM`
- `HV-SEM` -> `HV-SEM`
- `Fab` -> `팹`

### 1.2 번역 원칙

- `Overview`는 막연한 `개요`보다 운영 화면 의미가 분명한 `전체 현황`을 우선 사용합니다.
- `Inventory`는 재고 의미로 오해되지 않도록 이 프로젝트에서는 `장비 현황`을 기본값으로 사용합니다.
- `Online` / `Offline`은 `활성`, `비활성`보다 설비 연결 상태가 바로 보이는 `온라인`, `오프라인`을 사용합니다.
- 탭, 카드, 버튼, 표 제목이 서로 다른 표현을 쓰지 않도록 같은 개념은 동일한 용어를 유지합니다.

## 3. 화면 문구 작성 규칙

### 3.1 메뉴/탭/페이지 제목

- 제목은 짧고 단정하게 작성합니다.
- 현업 사용자가 스캔하기 쉬운 명사형을 우선 사용합니다.
- 영어와 한국어를 섞어야 할 때는 핵심 장비명은 유지하고, 의미 설명 부분만 한국어로 바꿉니다.

권장 예시:

- `CD-SEM 전체 현황`
- `장비 현황`
- `팹 선택`
- `보관 현황`

비권장 예시:

- `CD-SEM Overview`
- `Tool Inventory`
- `FAB Select`
- `Storage Dashboard`

### 3.2 버튼/링크

- 동작이 보이도록 동사형을 사용합니다.
- 링크처럼 보이는 작은 텍스트보다 버튼 라벨이 더 분명해야 합니다.
- 가능하면 사용자가 클릭 결과를 바로 예측할 수 있게 씁니다.

권장 예시:

- `상세 보기`
- `장비 검색`
- `필터 초기화`
- `새로고침`

비권장 예시:

- `보기`
- `이동`
- `Click here`

### 3.3 상태 배지/카운트

- 상태는 짧고 즉시 구분 가능해야 합니다.
- 배지 색상에만 의존하지 말고, 텍스트만 봐도 의미가 전달되어야 합니다.

권장 예시:

- `온라인`
- `오프라인`
- `사용 가능`
- `사용 불가`
- `N/A`

## 4. UI 친화성 기준

이 저장소에서는 번역 자체보다도 "현장 엔지니어가 빠르게 이해하고 조작할 수 있는가"가 더 중요합니다.

따라서 문구를 정할 때 다음을 함께 확인합니다.

- 사이드바, 탭, 카드, 표 제목이 같은 개념을 서로 다른 단어로 부르지 않는지 확인합니다.
- 클릭 가능한 요소는 본문 텍스트보다 버튼이나 명확한 액션 라벨로 보이게 합니다.
- 다크 모드와 라이트 모드 모두에서 배지/버튼 대비가 충분한지 확인합니다.
- 긴 영어 문장을 억지로 번역하기보다, 짧고 명확한 한국어 문구로 다시 씁니다.
- 표가 길어지면 검색, 페이지네이션, 요약 정보를 함께 제공해 탐색 부담을 줄입니다.

## 5. 이 프로젝트에서 바로 적용할 기본 매핑

현재까지 우선 적용할 기본 표현은 다음과 같습니다.

| 원문 | 기본 표현 |
| --- | --- |
| FAB | 팹 |
| Tool Inventory | 장비 현황 |
| CD-SEM Overview | CD-SEM 전체 현황 |
| Online | 온라인 |
| Offline | 오프라인 |

추가 용어가 생기면 이 문서에 계속 누적해서 같은 기준을 유지합니다.
